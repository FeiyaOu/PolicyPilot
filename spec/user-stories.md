# 用户故事与验收标准

## 1. 文档目的

本文档用于把 `project-architecture.md` 中的产品设想拆成可实现、可测试的用户故事。

本项目采用 SDD → TDD 的方式推进：

1. 先定义用户如何使用系统。
2. 再定义每个页面的验收标准。
3. 然后把稳定、可判断的行为转成测试用例。
4. 最后再进入代码实现。

## 2. 目标用户

### 2.1 银行客户经理

客户经理需要快速查询内部政策，例如考核规则、投诉处理、评聘申报、扣分规则等。

目标：获得准确、可追溯、有来源页码的答案。

### 2.2 分行运营/合规人员

运营或合规人员需要确认知识库是否完整、是否存在冲突、是否有过期政策。

目标：维护一个可信、可更新、可评估的政策知识库。

### 2.3 面试官/技术评审者

面试官需要看到本项目不是简单的 LLM 聊天页面，而是一个完整的 RAG 系统。

目标：能清楚看到 Query 改写、多路召回、BM25、向量检索、混合检索、Rerank、引用溯源、知识库健康检查、版本评估等能力。

## 3. 应用页面总览

第一版 Streamlit 应用包含以下页面：

1. 知识库构建
2. 政策问答助手
3. 检索实验室
4. 知识库健康检查
5. 知识库版本对比
6. 评估报告
7. 系统设置

## 4. 页面一：知识库构建

### 4.1 页面目标

用户可以从本地 PDF 政策文件构建 RAG 知识库。

第一版默认使用已有银行政策 PDF，后续支持添加更多 PDF。

### 4.2 用户故事

作为银行知识库维护人员，
我希望选择或使用默认的政策 PDF 文件，
然后一键构建向量索引和 BM25 索引，
以便后续可以进行政策问答和检索实验。

### 4.3 页面元素

- 当前数据目录显示
- 已发现 PDF 文件列表
- PDF 上传入口
- `构建知识库` 按钮
- 构建状态提示
- 文档处理结果
- Chunk 统计信息
- 索引保存路径
- 错误或警告信息

### 4.4 验收标准

- 当数据目录中存在 PDF 文件时，页面应显示 PDF 文件名。
- 用户应可以通过页面上传 PDF 文件。
- 用户上传 PDF 后，系统应将文件纳入本次知识库构建流程。
- 当用户点击 `构建知识库` 时，系统应执行文本抽取、切分、Embedding、FAISS 建库、BM25 建库。
- 构建完成后，页面应显示处理文档数、chunk 数、Embedding 模型名称、FAISS 索引路径、BM25 索引状态。
- 如果某些 PDF 页面无法抽取文本，页面应显示警告，但不应导致整个流程崩溃。
- 如果没有 PDF 文件，页面应提示用户添加文件或检查数据目录。
- 构建完成后，后续页面应能检测到知识库已可用。

### 4.5 可转化为测试的行为

- 给定一个测试 PDF，构建流程应返回非空 chunk 列表。
- 每个 chunk 应包含 `chunk_id`、`source_file`、`page_number`、`content`。
- 如果 PDF 某页无文本，构建结果应包含 warning。
- FAISS 和 BM25 构建成功后，应返回统一的索引状态对象。

## 5. 页面二：政策问答助手

### 5.1 页面目标

用户可以用自然语言提问，系统返回基于政策原文的答案，并展示引用来源。

### 5.2 用户故事

作为客户经理，
我希望输入一个政策问题，
系统能根据内部制度回答，并告诉我答案来自哪个 PDF 和第几页，
以便我可以信任并核对答案。

### 5.3 页面元素

- 问题输入框
- 可选对话上下文输入区
- 检索模式选择
- 是否启用 Query 改写
- 是否启用多查询召回
- 是否启用 Rerank
- `开始回答` 按钮
- 最终答案区
- 引用来源区
- RAG 过程展开区

### 5.4 验收标准

- 当用户输入问题并点击 `开始回答` 时，系统应返回答案或明确说明证据不足。
- 答案必须基于检索到的上下文生成。
- 答案区域必须显示引用来源，包括 PDF 文件名和页码。
- 如果启用 Query 改写，页面应显示原始问题和改写后问题。
- 如果启用多查询召回，页面应显示生成的查询变体。
- Rerank 作为高级选项，默认不强制开启。
- 如果用户启用 Rerank，页面应显示最终入选 chunk 的 rerank 分数。
- 页面应展示最终用于生成答案的证据 chunk。
- 回答页面只展示检索过程，不展示完整 Prompt。
- 当知识库尚未构建时，页面应提示先构建知识库。

### 5.5 可转化为测试的行为

- 给定问题和 mock 检索结果，答案生成输入应包含检索上下文。
- 引用列表应从 chunk metadata 中生成，而不是从 LLM 文本中解析。
- 当检索结果为空时，系统应返回证据不足提示。
- Query 改写结果应保留原始问题和改写问题。
- 多查询结果应去重。

## 6. 页面三：检索实验室

### 6.1 页面目标

让面试官和开发者直观看到不同检索策略的效果差异。

### 6.2 用户故事

作为技术评审者，
我希望用同一个问题比较不同检索策略的结果，
以便判断系统是否真正实现了向量检索、BM25、混合检索、多查询召回和 Rerank。

### 6.3 页面元素

- 测试问题输入框
- Top-K 参数
- Hybrid alpha 参数
- 检索策略多选框
- `运行对比` 按钮
- 多列结果展示
- 分数展示
- 来源页码展示
- Chunk 重叠分析

### 6.4 检索策略

第一版应支持：

- 向量检索
- BM25 检索
- 混合检索
- 混合检索 + 多查询
- 混合检索 + 多查询 + Rerank

### 6.5 验收标准

- 用户可以选择至少两种检索策略进行对比。
- 每种策略都应返回 top-k chunk、分数和来源页码。
- 混合检索应展示 BM25 分数、向量分数和融合分数。
- 多查询召回应展示查询变体和去重后的候选数量。
- Rerank 策略应展示 rerank 前后的排序变化。
- 页面应标记最终推荐用于问答的上下文。

### 6.6 可转化为测试的行为

- Hybrid 分数应按配置的 alpha 计算。
- 多查询合并结果应按 `chunk_id` 去重。
- Rerank 后结果应按 rerank score 降序排列。
- Top-K 参数应限制最终返回数量。

## 7. 页面四：知识库健康检查

### 7.1 页面目标

检查知识库是否存在覆盖不足、过期信息、冲突信息等问题。

### 7.2 用户故事

作为合规或运营人员，
我希望系统自动检查知识库质量，
指出缺失、过期和冲突的政策内容，
以便我可以维护一个可信的知识库。

### 7.3 页面元素

- 健康检查启动按钮
- 测试问题集选择
- 总体健康分
- 覆盖率分析
- 时效性分析
- 冲突分析
- 改进建议

### 7.4 验收标准

- 用户点击 `运行健康检查` 后，系统应生成健康检查报告。
- 报告应包含总体健康分。
- 报告应包含缺失知识、过期知识、冲突知识三个部分。
- 每个问题项应包含问题描述、严重程度和建议处理方式。
- 如果 LLM 返回结果无法解析，系统应给出结构化 fallback，而不是直接报错。

### 7.5 可转化为测试的行为

- 健康报告应符合固定 schema。
- 总体健康分应在 0 到 1 之间。
- 缺失、过期、冲突三个字段即使为空也必须存在。
- LLM JSON 解析失败时，应返回 fallback report。

## 8. 页面五：知识库版本对比

### 8.1 页面目标

展示知识库在不同版本之间的变化，以及变化是否改善检索效果。

第一版保留该页面入口，但页面内容显示“开发中”。完整版本对比能力放到第二阶段实现。

### 8.2 用户故事

作为知识库维护人员，
我希望比较两个知识库版本，
看到新增、删除、修改的内容，
并查看新版本是否提升了检索表现，
以便决定是否采用新版本。

### 8.3 页面元素

- 版本选择器
- 版本基础信息
- 新增 chunk 列表
- 删除 chunk 列表
- 修改 chunk 列表
- 检索表现对比
- 版本推荐结论

### 8.4 验收标准

- 第一版页面入口应存在。
- 第一版页面应明确显示“开发中”，并说明该能力将在第二阶段实现。
- 用户可以选择两个版本进行对比。
- 系统应展示新增、删除、修改、未变化的 chunk 数量。
- 系统应展示两个版本在测试问题上的表现差异。
- 系统应给出推荐结论，例如是否建议使用新版本。
- 如果版本不存在，页面应显示明确错误提示。

### 8.5 可转化为测试的行为

- 相同 `chunk_id` 但内容变化时，应识别为 modified。
- 只存在于新版本的 chunk 应识别为 added。
- 只存在于旧版本的 chunk 应识别为 removed。
- 版本性能比较应返回 accuracy、average retrieval time、pass rate。

## 9. 页面六：评估报告

### 9.1 页面目标

用固定测试集评估 RAG 系统，形成可复现的质量报告。

### 9.2 用户故事

作为开发者，
我希望运行一组标准政策问题，
查看检索命中率、引用完整性和回答质量，
以便在修改 RAG pipeline 后发现回归问题。

### 9.3 页面元素

- 测试集选择
- 检索模式选择
- `运行评估` 按钮
- 单题结果表格
- 总体指标
- 失败案例分析

### 9.4 验收标准

- 第一版先使用手写的 10 个评估问题。
- 后续版本再支持自动生成评估问题。
- 系统应能加载测试问题集。
- 每个测试问题应包含 query、expected_keywords 或 expected_source。
- 评估结果应包含每题 pass/fail。
- 总体指标应包含 recall@k、citation_rate、pass_rate。
- 失败案例应展示问题、实际检索 chunk、缺失原因。

### 9.5 可转化为测试的行为

- 当 expected_source 出现在 top-k chunk 中时，source recall 应为通过。
- 当答案包含引用来源时，citation check 应为通过。
- pass_rate 应等于通过题数除以总题数。
- 空测试集应返回明确错误。

## 10. 页面七：系统设置

### 10.1 页面目标

集中展示和配置系统运行参数。

### 10.2 用户故事

作为开发者，
我希望在页面中看到当前模型、索引路径、检索参数和环境检查结果，
以便快速确认系统是否可运行。

### 10.3 页面元素

- LLM 模型名称
- Embedding 模型名称
- Rerank 模型名称
- 数据目录
- 索引目录
- 默认 chunk size
- 默认 chunk overlap
- 默认 top-k
- 默认 hybrid alpha
- API Key 状态检查

### 10.4 验收标准

- 页面应显示当前配置值。
- 页面不应显示完整 API Key。
- 如果 API Key 缺失，页面应显示明确提示。
- 参数修改后，应影响后续流程或提示需要重建索引。

### 10.5 可转化为测试的行为

- API Key 状态只能显示存在或缺失，不能输出真实密钥。
- 配置对象应包含默认值。
- 修改 chunk 参数后，应标记索引需要重建。

## 11. 全局验收标准

- 应用 UI 使用中文。
- README 使用中文。
- 用户在知识库未构建时不能直接执行依赖索引的功能。
- 所有 RAG 答案都应展示引用或证据不足提示。
- 所有页面都应优先展示用户能理解的错误信息，而不是 Python traceback。
- 核心逻辑应位于 `src/`，Streamlit 只负责 UI 调用。
- 可测试逻辑应尽量不依赖 Streamlit。

## 12. 第一阶段优先级

第一阶段先实现最小可展示闭环：

1. 知识库构建
2. 政策问答助手
3. 检索实验室
4. 基础评估报告

第二阶段再实现：

1. 知识库健康检查
2. 知识库版本对比
3. 对话知识沉淀
4. 部署与 README 展示优化

## 13. 已确认产品决策

1. 第一版保留 `知识库版本对比` 页面入口，但页面显示“开发中”。
2. 第一版支持用户上传 PDF，同时也可以读取默认数据目录中的 PDF。
3. Rerank 作为高级选项，不作为默认强制流程。
4. 回答页面只展示检索过程，不展示完整 Prompt。
5. 第一版先手写 10 个评估问题，后续版本再支持自动生成评估问题。
